# 목장 주문 공유 + 쿠폰 발급/사용 + 알림 개선

## Context

길을여는교회 이음카페 주문 앱에 세 가지를 적용한다.

1. **목장 주문 공유**: 현재 "최근주문"(`/orders/history`)은 본인 주문 + 이름·목장이 **모두 일치**하는 주문만 보여준다. 같은 목장 교인끼리는 누가 주문했든 그 목장의 모든 주문 내역을 함께 볼 수 있게 한다.
2. **쿠폰 발급/사용**: 관리자가 "기타" 탭에서 회원별·목장별 쿠폰을 발급(할인율 5%~100% 자유 설정, 1회용·무기한)하고, 고객이 주문 시 쿠폰을 적용해 할인받을 수 있게 한다.
3. **알림 개선**: (a) 같은 목장 주문이면 그 목원 전체에게 알림이 가도록, (b) 매 이벤트마다 알림이 중복(OS 푸시 + 인앱 토스트/소리)으로 오는 문제를 제거한다.

DB는 Supabase CLI로 마이그레이션을 만들어 적용한다.

확정된 정책:
- 목장 공유: **목장 전체 공유** (같은 목장이면 그 목장 모든 주문 열람·알림)
- 그룹 쿠폰: **목장 단위 쿠폰** (목장에 1개 발급, 그 목장 회원 누구나 사용)
- 사용 정책: **1회용 + 무기한** (한 번 사용되면 소진, 만료일 없음)

## Part 1 — 목장 전체 주문 공유

### 변경: `app/routes/api.orders.history.tsx`
고객 분기(현재 `customer_name` + `church_group` 둘 다 일치로 조회하는 부분, 약 260~277행)를 **목장(`church_group`)만 일치**하는 조회로 변경한다.

- 기존: `.eq("customer_name", customerName).eq("church_group", churchGroup)`
- 변경: `church_group`이 존재하면 `.eq("church_group", churchGroup)` 만으로 조회 → 해당 목장 전체 주문을 가져와 본인 주문과 `mergeOrdersById`로 병합.
- `customerName` 조건 제거. `matchByProfile` 플래그는 `Boolean(churchGroup)` 기준으로 유지.
- service role 조회이므로 RLS 변경 불필요(서버에서 필터링). 클라이언트 `orders.history.tsx`는 그대로 동작(주문에 `customer_name` 표시되어 누구 주문인지 구분 가능).

목장 미설정(null) 고객은 기존처럼 본인 주문만 보인다.

## Part 2 — 쿠폰 기능

### 2.1 DB 스키마 (신규 `coupons` 테이블 + `orders` 컬럼 추가)

신규 테이블 `coupons`:
- `id uuid pk default gen_random_uuid()`
- `discount_percent int not null check (discount_percent between 5 and 100)`
- `target_type text not null check (target_type in ('user','group'))`
- `target_user_id uuid null references users(id) on delete cascade`
- `target_church_group text null`
- `description text null`
- `is_active boolean not null default true`  ← 1회용: 사용 시 false
- `used_at timestamptz null`, `used_by_user_id uuid null`, `used_by_order_id uuid null references orders(id) on delete set null`
- `created_by uuid references users(id)`, `created_at timestamptz default now()`
- 만료일 컬럼은 두지 않음(무기한). 인덱스: `(target_user_id)`, `(target_church_group)`, `(is_active)`.

`orders` 컬럼 추가:
- `coupon_id uuid null references coupons(id) on delete set null`
- `discount_amount numeric not null default 0`
- `total_amount`은 **할인 적용 후 최종 금액**으로 저장(보고서/매출 일관). 원금은 `order_items` 합으로 도출 가능, 필요 시 `discount_amount`로 역산.

RLS (`fix_rls_production.sql`의 `is_admin()` 패턴 재사용):
- 관리자: 전체 SELECT/INSERT/UPDATE/DELETE (`public.is_admin(auth.uid())`).
- 고객 SELECT: `is_active = true AND (target_user_id = auth.uid() OR target_church_group = (select church_group from users where id = auth.uid()))`.
- 쿠폰 발급/사용 처리는 서버(service role)에서 수행하므로 고객 INSERT/UPDATE 정책은 불필요.

### 2.2 Supabase CLI로 마이그레이션 적용
프로젝트 ref: `lfaayzaknwlcupaoqnbu` (env.example 기준). CLI v2.95.4 설치됨, `supabase/` 디렉터리 없음.

1. `supabase init` — `supabase/config.toml` 생성.
2. `supabase login` (또는 `SUPABASE_ACCESS_TOKEN`) 후 `supabase link --project-ref lfaayzaknwlcupaoqnbu` — DB 비밀번호 필요. **이 단계는 사용자 인증 정보가 필요하므로 진행 전 확인.**
3. `supabase migration new add_coupons` → 생성된 `supabase/migrations/<ts>_add_coupons.sql`에 위 스키마 + RLS 작성.
4. `supabase db push` 로 원격 적용.
5. (대안) 인증이 막히면 동일 SQL을 Supabase Dashboard SQL Editor에 직접 실행하도록 안내. 마이그레이션 파일은 그대로 리포에 남겨 둠.

기존 루트 SQL 파일들은 그대로 두고, 신규 변경만 `supabase/migrations/`로 관리(점진 전환).

- `Coupon` 타입(위 컬럼) 추가.

### 2.4 DB 유틸: `app/lib/database.ts`

- `getCouponsForUser(userId, churchGroup, client)` — 주문 화면용. `is_active=true` 이고 본인 또는 본인 목장 대상 쿠폰.

- `createOrder` 확장: 선택적 `coupon_id`, `discount_amount`를 받아 orders insert 시 저장. 호출은 orders.new 액션에서 검증 후 전달.

신규 라우트 `app/routes/admin.coupons.tsx`:

- 발급 폼: 대상 유형(회원/목장) 선택 → 회원이면 이름/이메일로 사용자 선택, 목장이면 목장명 입력(기존 distinct church_group 목록 제안), 할인율 슬라이더/입력(5~100), 설명(선택).
- 발급된 쿠폰 목록(대상, 할인율, 사용여부) 표시.
- 발급/조회는 `app/routes/api.coupons.tsx` (admin 전용 action/loader, `api.admin-orders.tsx`의 role 체크 패턴 재사용) 또는 `admin.coupons.tsx`의 loader/action에서 service role로 처리.

### 2.6 주문 시 쿠폰 적용: `app/routes/orders.new.tsx`
- **loader**: 로그인 사용자면 `getCouponsForUser`로 사용 가능 쿠폰 목록을 함께 반환.
- **UI**: 주문 검토(`OrderReviewSheet`) 또는 주문 폼에 "쿠폰 선택" 항목 추가. 선택 시 할인율 적용 후 금액 미리보기(`원금`, `할인`, `최종금액`).
  - 쿠폰 미적용: 기존처럼 총 결제금액 1개만 표시.
  - 쿠폰 적용: 기존 총액은 **붉은 글씨 + 가로 취소선**(`line-through text-red-600`)으로 표시하고, 그 아래/오른쪽에 **할인 후 최종 결제금액**을 굵게 표시.
  - 할인 정보는 `쿠폰 20% 적용`, `-1,000원 할인`처럼 별도 행으로 보여 사용자가 계산 결과를 즉시 확인할 수 있게 한다.
  - 하단 제출 버튼 문구도 쿠폰 적용 후 금액 기준으로 `₩{finalAmount} 주문하기`를 표시한다.
- **action (`intent='createOrder'`, 약 217~334행)**: 전달된 `couponId` 검증 →
  1. service role로 쿠폰 재조회: `is_active=true` & 대상(본인/본인목장) 일치 확인. 불일치/소진이면 에러 반환.
  2. `discountAmount = round(totalAmount * percent / 100)`, `finalAmount = totalAmount - discountAmount`.
  3. `createOrder({ ..., total_amount: finalAmount, coupon_id, discount_amount })`.
  4. 주문 생성 성공 후 `redeemCoupon(couponId, userId, order.id)` 호출(1회용 소진). 소진 실패(경쟁) 시 주문 롤백 또는 에러 처리.
- 금액 계산 기존 지점: `totalAmount = items.reduce(... total_price)` (약 260행) — 이 뒤에 할인 적용.

## Part 3 — 목장 알림 + 중복 제거

### 3.1 목장 전체에 알림 (목원 누가 주문/상태변경해도 목원 모두에게)
- **수정 `app/routes/api.admin-orders.tsx` `getNotificationRecipientIds`** (약 128~155행): 현재 `.eq("name", customerName).eq("church_group", churchGroup)` 로 **이름+목장 둘 다** 일치를 요구 → 사실상 목원에게 안 감. **목장(`church_group`)만 일치**로 변경해 그 목장 전 목원을 수신자로 포함(+ `order.user_id` 항상 포함, Set 중복 제거 유지). 목장 null이면 기존대로 주문자만.
- **수정 `app/lib/database.ts` `createOrder`의 새 주문 알림** (약 296~338행): 현재 주문자(order_confirmation) + 관리자/스태프(new_order)에게만 발송. 여기에 **같은 목장 목원**에게도 새 주문 알림(예: type `group_order`, "○○님이 주문했습니다")을 추가. 목원 조회는 `church_group` 일치(주문자 본인 및 관리자는 중복 제외). 위 `getNotificationRecipientIds`와 동일한 "목장 일치" 헬퍼로 공통화.

### 3.2 중복 알림 제거 (핵심)
근본 원인 4가지를 정리:

**(A) Foreground 중복 — `public/sw.js` push 핸들러.** 앱이 열려 활성화된 상태에서도 `self.registration.showNotification(...)`을 무조건 호출 → OS 푸시 + 인앱 토스트/소리가 동시에 발생.
- 수정: push 핸들러에서 `clients.matchAll({type:'window', includeUncontrolled:true})`로 **포커스(visibilityState==='visible')된 클라이언트가 있으면**, OS 알림을 띄우지 않고 `client.postMessage({type:'push', payload})` 로 전달만. 포커스 클라이언트가 없을 때만 `showNotification` 호출.
- 인앱(`NotificationContext`)은 이미 realtime로 토스트를 처리하므로, postMessage는 보조 경로(필요 시 무시 가능). 즉 **앱 열림 → 인앱 토스트만, 앱 닫힘 → OS 푸시만.**

**(B) 인앱 announce 채널 이중 — `app/contexts/NotificationContext.tsx`.** 새 주문에 대해 `notifications-toast-${userId}`(notifications INSERT)와 `orders-insert-audio-${userId}`(orders INSERT) **두 채널이 각각** 소리+토스트를 시도(현재 `shouldAnnounceOrder` 타이밍 가드로 겨우 1개만 통과하는 경쟁 구조).
- 수정: 새 주문 소리/토스트 소스를 **한 채널로 단일화**. `notifications` 채널의 `new_order` 분기를 단일 소스로 삼고 `orders-insert-audio` 채널 제거(또는 그 반대). `shouldAnnounceOrder` 가드는 안전망으로 유지.

**(C) 상태변경 알림 로직 이원화 — `app/lib/database.ts` `updateOrderStatus()`.** 이 함수는 호출처가 없는 **미사용(dead) 코드**이며 `api.admin-orders.tsx`의 상태 알림과 중복된 로직을 가짐. 혼선 방지를 위해 제거(상태변경 알림의 단일 소스 = `api.admin-orders.tsx`).

**(D) 푸시 tag 정리 — `public/sw.js` / 서버 push payload.** 같은 주문·이벤트는 동일 `tag`(`${orderId}:${type}`)로 보내 OS가 동일 알림을 덮어쓰게 하고 `renotify`는 상태변경 등 의미 있는 변경에만 true. (A) 수정과 함께 잔여 중복 방지.

> 구현 시작 시 systematic-debugging으로 위 (A)~(D) 중 실제 재현되는 중복을 1개씩 확인 후 수정. 가장 유력한 주범은 (A) foreground 푸시 + 인앱 토스트 이중.

## 검증
1. `npm run typecheck` 통과.
2. **목장 공유**: 같은 `church_group`을 가진 회원 2명 계정으로 각각 주문 → 둘 다 `/orders/history`에서 두 주문 모두 보이는지 확인. 목장 없는 회원은 본인 주문만 보이는지 확인.
3. **쿠폰 발급**: 관리자로 기타 탭 → 쿠폰 발급 → 회원 쿠폰/목장 쿠폰 각각 생성, 목록 표시 확인.
4. **쿠폰 사용**: 대상 회원으로 주문 → 쿠폰 선택 시 5%~100% 할인 반영, 기존 가격은 붉은 취소선으로 표시되고 계산 후 최종 금액이 강조 표시되는지 확인. 최종 금액 저장 확인. 사용 후 같은 쿠폰이 더 이상 목록에 안 뜨고 재사용 거부되는지 확인(1회용).
5. **DB**: `supabase db push` 적용 후 Supabase에서 `coupons` 테이블·`orders` 신규 컬럼·RLS 정책 존재 확인. (scripts/create-test-order.mjs 등으로 보조 검증 가능)
6. 매출 보고서(`/reports`)가 할인 적용된 `total_amount`로 정상 집계되는지 확인.
7. **목장 알림**: 같은 목장 회원 A가 주문 → 같은 목장 회원 B(앱 닫힘)는 OS 푸시, B(앱 열림)는 인앱 토스트 수신. 관리자도 새 주문 알림 수신.
8. **중복 제거**: 앱을 열어 둔 상태에서 주문/상태변경 발생 시 알림이 **1회만** 표시되는지 확인(OS 푸시와 인앱 토스트가 동시에 안 뜸). 앱을 닫은 상태에서는 OS 푸시 1회만. 새 주문 소리도 1회만.

## 주의/확인 필요
- `supabase link` / `db push`는 Supabase 로그인 + DB 비밀번호가 필요. 인증 정보 제공이 어려우면 마이그레이션 SQL을 Dashboard에서 수동 실행하는 방식으로 대체.
- 목장 단위 쿠폰 1회용 = 목장에서 **먼저 사용한 1명**이 소진. (회원마다 1회씩 쓰게 하려면 회원별 개별 발급 필요 — 추후 옵션.)
